# Handoff — PixlerPay / Paynix Commission Dashboard

Last updated: 2026-07-10

## What this project is

Automates commission tracking across two merchant payment platforms and
publishes a live, tabbed web dashboard:

- **PixlerPay** — logs into 18 individual merchant accounts, downloads
  payout reports, calculates commission as margin (Onboarded % − Reseller %)
  per the Google Sheet rate card.
- **Paynix** — logs into one reseller account (covers 12 merchants),
  scrapes commission (computed server-side by Paynix, no math needed),
  merchant wallet balances, and failed payouts.

Both feed a single live dashboard hosted on GitHub Pages, reading data from
Google Drive.

## One command to refresh everything

```powershell
npm run all
```

Runs, in order:
1. `download-report` — logs into each PixlerPay account in `data/accounts.json`, downloads payout CSVs into `data/`
2. `calculate` — matches transactions to `data/commission-rates.json`, writes `website/commission-results.json` (includes per-transaction log for date-range filtering)
3. `download-paynix` — logs into Paynix reseller portal, scrapes dashboard/merchants/failed-payouts, writes `website/paynix-results.json`, diffs against the previous snapshot for wallet changes / new failures
4. `upload-to-drive` — pushes both JSON files to Drive (OAuth, fully unattended). The dashboard picks up changes automatically on its next fetch (every 5 min) or via its Refresh buttons — no redeploy needed.

Individual steps can also be run separately (`npm run download-report`, `npm run calculate`, `npm run download-paynix`, `npm run upload-to-drive`).

## Architecture notes / decisions made along the way

- **Commission logic differs by platform:** PixlerPay commission = margin (Onboarded % − Reseller %) calculated by us; Paynix commission is read directly from their dashboard (they compute it).
- **PixlerPay reports have no VA/client column** — each login is already scoped to one client, so matching is by account→client name, not per-row VA.
- **Service accounts don't work for personal Gmail Drive uploads** — no storage quota outside Google Workspace Shared Drives. Switched to OAuth against the actual Google account instead (see `scripts/gdrive-oauth-setup.js`).
- **GitHub Pages requires a public repo** on the free plan (private repos need GitHub Pro for Pages) — repo was switched to public since data privacy wasn't a concern for this use case.
- **Artifact hosting (claude.ai) can't fetch external data** — its CSP blocks all outbound network requests, which is why the dashboard is hosted on GitHub Pages instead, not as a Claude Artifact.
- **Custom date-range filtering** works by shipping the full per-transaction log (`transactions: [[va, isoDate, amount], ...]`) to the browser, which recomputes aggregates client-side for whatever range is selected — not just the last-30-days snapshot.
- **Wallet-change / failed-payout detection** works by diffing each Paynix snapshot against the previous run's snapshot (`data/paynix-snapshot-previous.json`), not by scraping a change log.

## Known limitations / not-yet-done

1. **"Refresh" buttons on the dashboard only re-fetch the latest Drive snapshot** — they cannot trigger a brand-new live scrape on demand, because GitHub Pages is a static site with no backend. Genuine "click → scrape happens now" requires a real server holding credentials.
   - **Decision made:** build this on a cloud VM (not the user's own PC + tunnel).
   - **Status:** waiting on the user to provision a DigitalOcean (or similar) VM and share IP + SSH access. Nothing built yet on this front.
2. **Telegram alerts (wallet changes / failed transactions) — not yet built.**
   - **Decision made:** Telegram over WhatsApp (free, ~2 min setup vs. WhatsApp Business API friction).
   - **Status:** waiting on the user to create a bot via @BotFather and share the token + send the bot one message (needed to capture the chat ID). Nothing built yet on this front.
3. **N V CONNECT ACROSS PRIVATE LIMITED (PixlerPay)** — automation fails; its "DT and Payout" combined solution type has a different portal layout (the "Payouts Transaction" nav link isn't found). Not investigated further — deprioritized per user instruction to skip it.
4. **3 PixlerPay clients showed zero transactions** in the last scrape (Elleaura, FINFLEX, WESURE cybertech/infra/innovations) — flagged as worth a manual sanity check since they're marked Active/Onboarding in the rate sheet, but not investigated.
5. **Paynix "Added/Deducted" wallet columns are inferred from balance deltas** between runs, not from a real per-merchant topup transaction log (which would require visiting each of the 12 merchants' individual detail pages — not built). If a merchant's balance changes for a reason other than a topup/debit (e.g., a fee), it'll still show up in these columns since it's just "current − previous."
6. **Automation run cadence is manual only** — no Windows Task Scheduler job set up yet (was explicitly deferred by the user "manual only for now").

## Telegram bot — design notes (not yet built)

Captured 2026-07-10 while triaging the dashboard's failed-payout alert (see
below). Not implemented — needs the VM (item 1 above) as a host, since
Telegram alerting requires a long-running process, not a static site.

- **Reuse the rate-limit filter already built for the dashboard**: Paynix's
  "Please wait at least N minutes between transactions" gateway message is
  retry throttling, not a real failure — `download-paynix.js` now excludes
  it via `RATE_LIMIT_REASON_RE`. The bot must apply the same filter (or call
  the same scraper output) so it doesn't spam alerts for non-failures. This
  is also why a real failure category worth alerting on emerged once the
  noise was removed: "Low balance to make this request" (merchant wallet
  can't cover the payout) — genuinely alert-worthy.
- **24×7 servicing**: a Telegram bot polling/pushing on a schedule needs a
  process that's always running — this is the actual argument for the VM
  (item 1), not just on-demand refresh. Options once the VM exists:
  process manager (pm2 / systemd service) running the scrape+diff+alert
  loop on a cron-like interval, restarting on crash. Uptime monitoring
  (e.g. a dead-man's-switch ping to an external service) is worth adding
  so a silently-dead bot doesn't go unnoticed.
- **Per-merchant wallet history**: alerting meaningfully on wallet
  drops needs more than the current balance-delta-between-runs approach
  (see Known Limitation #5) — ideally a short rolling log (last 5 balance
  readings with timestamps) per merchant, scraped from each merchant's own
  wallet page, not just the aggregate merchants list. Not built; needs the
  VM's always-on scraper to accumulate history between runs rather than
  only keeping one "previous" snapshot.
- **Alert scope**: wallet drops, low-balance-driven failed payouts (see
  above), and any genuinely new failed payout (post rate-limit filtering).

## Refresh button — plan (not yet built)

Currently "Refresh" only re-fetches whatever snapshot last landed on Drive
(see Known Limitation #1) — it can't trigger a new scrape. Once the VM
backend exists, the intended behavior is:

1. Refresh button calls an authenticated endpoint on the VM (e.g.
   `POST /api/refresh`) instead of hitting Drive directly.
2. VM endpoint runs the same `npm run all` pipeline synchronously (or kicks
   it off and the frontend polls a `/api/status` endpoint for completion,
   since a full scrape across 18 PixlerPay logins + Paynix takes real time
   — polling is likely necessary rather than a single blocking request).
3. On completion, the endpoint uploads to Drive as today, then returns the
   fresh data (or a "done" signal) directly to the browser, so the
   dashboard shows the exact data generated by *this* click — not the
   previous 5-minute auto-poll's snapshot.
4. Show a spinner/progress state on the button while the scrape runs
   (could take 1-2+ minutes for PixlerPay's 18 accounts).
